2.1.2 Run Image

Introduction to environment variables and instructions for using commands.

  • GIT_ADDRESS

xxxxxxxxxx
The address of the GIT warehouse where the code is located.
xxxxxxxxxx
docker run -it --rm --privileged \
-v D:\out:/package \
-e GIT_ADDRESS=http://atc-gitlab.advantest.com/Ian/fusiaserver.git \
10.150.111.250/fusia/fusia_build_env_image:v4.3 \
bash fusia_package_config.sh

  • SOURCE_CODE

xxxxxxxxxx
Whether to get the source code when packaging and compiling.
The obtained source code will be placed in the directory specified by the -v parameter.
Source code is not fetched by default.
xxxxxxxxxx
docker run -it --rm --privileged \
-v D:\out:/package \
-e SOURCE_CODE=true \
-e GIT_ADDRESS=http://atc-gitlab.advantest.com/Ian/fusiaserver.git \
10.150.111.250/fusia/fusia_build_env_image:v4.3 \
bash fusia_package_config.sh

  • CODE_VERSION

xxxxxxxxxx
Specify the version of the code compiled and packaged.
By default, the latest version of the code from the master branch is packaged.
xxxxxxxxxx

3.Usage Instructions

3.1 Modify the service default configuration

Configuration file path: /opt/FuSIA/fusia/config/application.conf.

Before starting FuSIA, please modify the interface in the configuration file to the actual IP address of the current machine.

The default configuration is as follows:

xxxxxxxxxx
fusia-server {
    # Token
    token = "XYZ"


    # The sub-service registers itself on this node,For now, please use this as the address of the service and keep it the same as fusia-server-father
    root {
        interface = "0.0.0.0"
        port = 50051
    }


    # The address used by FuSIA by default, by default, generally as FuSIA's father node
    father {
        interface = "0.0.0.0"
        port = 50051
    }


    # Currently only supports configuration of local IP
    children {
       childrenNode = [
            {
               interfaces = "0.0.0.0"
               port = 50052

3.3 Starting the sub-services

Note: Before starting the sub service, make sure that session has been started.

xxxxxxxxxx
# starttesim --session [session_name]
starttesim --session pc-admin_1

  • Call the interface provided by the main service to start the sub-service.

xxxxxxxxxx
service ServerCommandService {
    /*
      {
        "value": [
            "session:session_name"
        ]
      }
    */
    rpc startFusia(OptionStrValue) returns(Result){
      option deprecated = true;
    }
}


message OptionStrValue{
  // values
  repeated string value = 1;
}


message Result{
  // Type of result
  ResultType type = 1;
  // return code
  int32 returnCode = 2;
  // output message after running commands
  string out = 3;
  // error message
  string err = 4;
  // trace info
  string traceInfo = 5;
}


enum ResultType{
  // unknown
  Unknown = 0;
  // success
  Success = 1;
  // error
  Error = 2;
}

  • Start sub-services via the gRPCurl command.

xxxxxxxxxx
grpcurl -H 'token:XYZ' \
-d '{"value":"session:session_name"}'  \
-plaintext 10.150.115.143:50051 proto.cmd.ServerCommandService.startFusia

example: 

xxxxxxxxxx
grpcurl -H 'token:XYZ' \
-d '{"value":"session:cell01"}'  \
-plaintext 10.150.115.143:50051 proto.cmd.ServerCommandService.startFusia

  • Start sub-services from the command line.

xxxxxxxxxx
fusia if:0.0.0.0 port:50052 session:session_name -DSESSION=session_name &

example:

xxxxxxxxxx
fusia if:0.0.0.0 port:50052 session:cell01 -DSESSION=cell01 &
xxxxxxxxxx
# Starting services in the Linux background
nohup fusia if:0.0.0.0 port:50052 session:cell01 -DSESSION=cell01 &
